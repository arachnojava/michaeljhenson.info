1 General Code Style

1.1 Indentation

Lines of code are indented by a four-space tab from the start of the block that contains them.

 

if (fileExists(filename))

    promptForOverwrite();

 

1.2 Braces

Opening curly braces are on a line by themselves and align with the header delimited by the block.  Closing braces align with opening braces.  (Note that this does not comply with official Java standards.  See References.)

 

for (DebrisChunk c : debrisList)

{

    c.update(elapsedTime);

    if (surfaceCollision(c))

    {

        playCollisionSound();

        c.setState(DebrisDeadState.getInstance());

    }

}

 

1.3 Line Widths

Lines of code should not exceed 80 characters in length.  Seventy is a good guideline because it leaves adequate space for handwritten annotations on a printed page.  If a line of code must be broken up, the lines following the first line should be indented at least twice.  Optionally, parameters or similar operations may be vertically aligned for neatness.

 

1.4 Declarations

1.4.1 Placement of Declarations

All declarations should follow the principle of locality.  That is, they should be accessible in a scope no broader than where they are required.  This simplifies the code and makes more efficient use of memory.

 

1.4.2 Number of Declarations per Line

Each declaration should be on its own line of code.  This makes them easier to locate quickly and makes it easier for developers to comment each one as they deem appropriate.

 

2 Naming Conventions

The naming conventions here comply with and extend the standard Java naming conventions.

 

2.1 Class-Level Naming Conventions

2.1.1 Classes

Classes should follow the standard Java naming convention.  (See Resources.)  Class names begin with a capital letter.  If a class name contains multiple words, the first letter of each word is capitalized.  Do not use underscores between words in a class name.

 

Classes in the MHFramework API begin with MH to distinguish them from classes that originate in the Java, Android, or other libraries.

2.1.2 Interfaces

Interface names follow the same convention as class names.

2.1.3 Enumerations

Enumerations follow the same convention as class and interface names.

 

2.2 Member Naming Conventions

2.2.1 Methods

Method names shall be descriptive of the methods purpose and behavior.

 

Method names shall begin with a lowercase letter, but the initial letter of each following word in the name shall be capitalized.  Do not use underscores between words in a method name.

 

public void setMapCellAddress(int row, int column)

2.2.2 Data Members

Data members shall follow the same naming conventions as methods.

 

private MHVector previousVelocity;

2.2.3 Constants

Constants shall be named in all capital letters with underscores between words.

 

static final int MIN_SIGHT_RANGE = 100;

2.2.4 Inner Classes

Named inner classes shall follow the same naming convention as regular, top-level classes.

 

Anonymous inner classes are generally avoided.

 

2.3 Local Naming Conventions

2.3.1 Local Variables

Local variable names follow the same conventions as data members.

2.3.2 Parameters

Parameters must have a descriptive name to disambiguate their purpose.

 

Parameters names follow the same conventions as local variables and data members.

 

3 Comment Styles

3.1 Class Header Comments

Each class, interface, and enumeration should have a header comment that follows Javadoc conventions.  (See References.)  At a minimum, the comment should contain one descriptive sentence indicating the purpose and responsibilities of the class.  If the class is a vital hook into the engine, it should also contain instructions or sample code to assist developers in its use.

3.2 Method Header Comments

Every non-private method should include a header comment explaining the purpose of the method, the significance of any parameters passed in, and the meaning of the returned value if one exists.  Javadoc provides tags for communicating parameters and returned values.  (See Reference.)

3.3 Package Comments

Each package shall contain a package-info.java file that summarizes the purpose of the package in Javadoc notation.  (See References.)

 

3.4 Optional Code Block Comments

In code blocks that are particularly long or deeply nested, it can be difficult to tell which block is ending with each closing brace, so it is helpful to add an end-of-line comment with the keyword and/or conditional statement that began the block.  As noted in the title of this section, this practice is optional and situational.
